## 🎯 Бізнес-задача

Ми навчились розгортати наш Flask-додаток як надійний systemd-сервіс. Але додаток сам по собі не дуже корисний без бази даних. Уявімо, що ми додаємо до нашого `training-project` можливість зберігати дані у PostgreSQL.

Стандартний шлях: встановити PostgreSQL вручну. Однак це той самий «Pets»-підхід, від якого ми пішли в Темі 6:

- Потрібно вручну встановити пакети, ініціалізувати кластер, створити користувача і базу.
- На кожній новій VM — робити все заново.
- Якщо версія PostgreSQL відрізняється — можливі несумісності.
- Бібліотеки і залежності PostgreSQL можуть конфліктувати з іншими пакетами на системі.

**Нам потрібен стандартний, відтворюваний спосіб запуску допоміжних сервісів (PostgreSQL, Redis, тощо) без «забруднення» хост-системи.** Щоб одна команда замінила десять кроків ручного налаштування — і працювала однаково на будь-якій VM.

У більш широкому сенсі ця проблема стосується не лише баз даних. Класична скарга розробника — **«У мене на машині все працює!»** — виникає через відмінності в середовищах. Docker вирішує це через стандартизований спосіб **упакування і запуску** програм. У цій темі ми побачимо обидва сценарії: запуск готового стороннього сервісу (PostgreSQL) та принцип створення власних образів (Dockerfile).

**Зв'язок з проєктом:** У `devops-ai-assistant/` є `docker-compose.yml` — саме так реальний AI-асистент запускає свої інфраструктурні залежності (базу даних, допоміжні сервіси). Це не навчальне спрощення — це виробничий стандарт.

## 🧠 Необхідні знання

### 1. Проблема «у мене на машині працює» — і чому це системна проблема

Уявіть типову ситуацію командної розробки:

- Розробник А пише код на macOS, Python 3.11, Flask 3.0.
- Розробник Б підключається, у нього Ubuntu, Python 3.9, Flask 2.3.
- Production-сервер — Ubuntu 22.04, Python 3.10.
- Сервер замовника — CentOS 7, Python 3.6.

Щоразу виникають несумісності: деякі API в Python змінились між версіями, якийсь пакет веде себе по-різному, конфігурація systemd відрізняється. Команда витрачає час не на розробку функцій, а на налагодження середовища.

Ця проблема стара, як програмування. Різні спроби вирішення:

| Рішення                        | Ідея                                                                           | Обмеження                                                                      |
| ------------------------------------- | ---------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| «Пишіть інструкцію» | README з кроками налаштування                                  | Застаріє наступного тижня. Людський фактор         |
| Virtualenv (Python)                   | Ізолювати Python-залежності                                     | Вирішує лише Python. ОС, системні бібліотеки — ні     |
| Vagrant (VM)                          | Повна VM для кожного розробника                           | Забирає 1-2 ГБ RAM, довго стартує, важко поширювати |
| **Docker (контейнер)** | Запакувати додаток + залежності в один об'єкт | Потрібен Docker Engine; слабша ізоляція ніж VM                 |

Docker вирішує цю проблему через стандартизований спосіб **упакування і запуску** додатків.

### 2. Контейнер vs. Віртуальна машина — правильне порівняння

У Темі 1 ми свідомо обрали VM, а не Docker, бо нам потрібен повноцінний `systemd`. Тепер — давайте розберемось, чому для деяких задач контейнер краще.

#### Що відбувається всередині

**Віртуальна машина (VirtualBox, QEMU):**

```
┌────────────────────────────────────────┐
│         Ваш додаток                    │
│         Python, Flask, залежності      │
├────────────────────────────────────────┤
│         Гостьова ОС (Ubuntu 22.04)     │
│         systemd, ядро Linux, драйвери  │
├────────────────────────────────────────┤
│         Гіпервізор (VirtualBox)        │
├────────────────────────────────────────┤
│         Хост-ОС                        │
│         Хардвер                        │
└────────────────────────────────────────┘
```

VM емулює повноцінний комп'ютер з власним ядром ОС. Це дає максимальну ізоляцію, але за ціною: кожна VM забирає сотні МБ лише для ОС.

**Контейнер (Docker):**

```
┌──────────────┐  ┌──────────────┐
│  Контейнер 1 │  │  Контейнер 2 │
│  (Flask-app) │  │  (PostgreSQL)│
└──────────────┘  └──────────────┘
         │                │
┌────────────────────────────────┐
│      Docker Engine             │
│   (використовує ядро хоста)    │
├────────────────────────────────┤
│      Хост-ОС (Ubuntu)          │
│      Ядро Linux                │
│      Хардвер                   │
└────────────────────────────────┘
```

Контейнер **не емулює ОС**. Він використовує ядро хост-системи, але ізолює процеси через механізми Linux: **namespaces** (ізоляція простору імен) та **cgroups** (обмеження ресурсів). Ті самі механізми, які systemd використовує для ізоляції сервісів — тільки більш глибокі.

#### Практичне порівняння

| Характеристика | Віртуальна машина           | Контейнер (Docker)                           |
| ---------------------------- | ------------------------------------------- | ----------------------------------------------------- |
| Час запуску        | 30–60 секунд                         | Менше секунди                             |
| Розмір образу    | 1–10 ГБ                                  | 10–500 МБ                                          |
| Використання RAM | 256 МБ–2 ГБ (тільки для ОС) | Лише те, що потрібно додатку   |
| Ізоляція             | Повна (окреме ядро)          | Процесна (спільне ядро хоста) |
| systemd всередині   | ✅ Так (повна init-система)  | ❌ Зазвичай ні (один процес)      |
| Портативність   | Складно (великі файли)    | Легко (Docker Hub, реєстри)               |

> **Про systemd у VM vs. контейнері:** У Темі 1 ми обрали VM саме тому, що нам потрібен systemd для Теми 7 (управління сервісами). Контейнер зазвичай запускає один процес напряму — без init-системи. Це не недолік, а інша модель: контейнер — не "маленька VM", а "ізольований процес".

**Важливий висновок:** контейнер — це не «маленька VM». Це інструмент з іншою областю застосування. VM і Docker не конкурують — вони доповнюють одне одного. Ми часто запускаємо Docker **всередині** VM (саме так у цьому курсі).

**Ключові поняття:**

- **Контейнер** — ізольований процес на хост-системі; використовує ядро хоста, не містить власного ядра ОС
- **Docker Engine** — програма, що створює та керує контейнерами на хост-системі

### 3. Ключові концепції Docker: образ, контейнер, реєстр

Docker оперує трьома фундаментальними концепціями, які важливо чітко розрізняти.

#### Образ (Image) — рецепт

**Docker Image** — це незмінний snapshot файлової системи та конфігурації. Думайте про нього як про рецепт страви або клас в об'єктно-орієнтованому програмуванні: сам по собі він нічого не «робить», але з нього можна створити будь-яку кількість екземплярів.

Образ складається з **шарів (layers)**. Кожна інструкція в `Dockerfile` додає новий шар.

```
┌────────────────────────────────┐  ← Шар 4: наш app.py та requirements.txt
├────────────────────────────────┤  ← Шар 3: pip install flask
├────────────────────────────────┤  ← Шар 2: python:3.11-slim
├────────────────────────────────┤  ← Шар 1: debian:bookworm-slim (базовий образ)
└────────────────────────────────┘
```

**Переваги шарової архітектури — повторне використання шарів:**

Уявімо, що ви створюєте два образи:

**Flask-додаток:**

```
Шар 4: app.py + requirements.txt
Шар 3: pip install flask
Шар 2: python:3.11-slim
Шар 1: debian:bookworm-slim
```

**Django-додаток:**

```
Шар 4: manage.py + requirements.txt
Шар 3: pip install django
Шар 2: python:3.11-slim          ← той самий!
Шар 1: debian:bookworm-slim      ← той самий!
```

Обидва образи використовують `FROM python:3.11-slim` — тому Шари 1 і 2 **спільні**. Docker зберігає їх один раз на диску і використовує для обох образів:

```
/var/lib/docker/overlay2/
  ├── abc123... (debian:bookworm-slim)  ← спільний шар
  ├── def456... (python:3.11-slim)      ← спільний шар
  ├── ghi789... (pip install flask)     ← тільки для Flask
  ├── jkl012... (app.py Flask)          ← тільки для Flask
  ├── mno345... (pip install django)    ← тільки для Django
  └── pqr678... (manage.py Django)      ← тільки для Django
```

Замість 8 шарів (4 + 4) Docker зберігає лише **6 шарів** (2 спільних + 4 унікальних). Якщо у вас 10 образів на базі `python:3.11-slim`, базові шари зберігаються один раз — економія десятків гігабайт.

#### Контейнер (Container) — запущений екземпляр образу

**Контейнер** — це образ у стані «запущено». З одного образу можна запустити десятки контейнерів одночасно — кожен зі своїм ізольованим процесом, мережею та (опційно) власними даними.

Аналогія: образ — це клас, контейнер — це об'єкт (екземпляр класу). Або: образ — це ISO-файл інсталятора, контейнер — це встановлена і запущена система.

#### Реєстр (Registry) — сховище образів

**Docker Hub** — публічний реєстр образів. Там є офіційні образи для PostgreSQL, Redis, Nginx, Python, Node.js та тисяч інших інструментів. Ми можемо завантажити готовий образ і запустити його без написання єдиного рядка конфігурації.

```bash
# Завантажити образ PostgreSQL 16 з Docker Hub
docker pull postgres:16

# Запустити контейнер з PostgreSQL
docker run postgres:16
```

**Ключові поняття:**

- **Docker Image (образ)** — незмінний шаблон файлової системи; «рецепт» для контейнера
- **Docker Container (контейнер)** — запущений екземпляр образу; живий, ізольований процес
- **Layer (шар)** — один крок у побудові образу; спільні шари зберігаються один раз
- **Registry (реєстр)** — сховище Docker-образів; може бути публічним або приватним
- **Docker Hub** — найбільший публічний реєстр; містить офіційні образи postgres, nginx, python тощо

### 4. Dockerfile — як описати власний образ

У лабораторній ми будемо використовувати **готовий** образ PostgreSQL з Docker Hub. Але для розуміння Docker важливо знати, як створюються образи — бо саме це ви робитимете у виробничих проєктах.

`Dockerfile` — це текстовий файл з інструкціями для побудови Docker-образу. Це Infrastructure as Code на рівні одного додатку: замість «встановіть Python, потім flask, потім скопіюйте файли» — декларативні інструкції, які виконуються однаково на будь-якій машині.

#### Приклад: Dockerfile для Flask-додатку

> **Про `requirements.txt`:** У Темі 6 ми створили цей файл командою `echo "flask" > requirements.txt`. Це стандартний спосіб Python-проєктів перераховувати залежності — одна бібліотека на рядок. Команда `pip install -r requirements.txt` встановлює все, що перелічено у файлі. Для нашого Flask-додатку там один рядок: `flask`.

```dockerfile
# 1. Базовий образ: офіційний Python 3.11, мінімальна версія (slim)
FROM python:3.11-slim

# 2. Встановити робочу директорію всередині контейнера
WORKDIR /app

# 3. Спочатку скопіювати тільки requirements.txt (для кешування шарів)
COPY requirements.txt .

# 4. Встановити залежності (цей шар кешується, якщо requirements.txt не змінився)
RUN pip install --no-cache-dir -r requirements.txt

# 5. Тепер скопіювати весь код
COPY . .

# 6. Оголосити порт, на якому слухає додаток (документація, не фаєрвол)
EXPOSE 5000

# 7. Команда, яка запускається при старті контейнера
CMD ["python", "app.py"]
```

#### Основні інструкції Dockerfile

| Інструкція | Призначення                                                                                                                            |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| `FROM`             | Базовий образ. Обов'язкова перша інструкція                                                                  |
| `WORKDIR`          | Визначає робочу директорію для наступних команд                                                         |
| `COPY`             | Копіює файли з хост-системи в образ                                                                                  |
| `RUN`              | Виконує команду під час**збірки** образу (результат фіксується як шар)              |
| `CMD`              | Команда за замовчуванням при**запуску** контейнера (може бути перевизначена) |
| `EXPOSE`           | Документує порт (не відкриває порт автоматично — це робить `-p` при `run`)                |
| `ENV`              | Встановлює змінну оточення в образі                                                                                |

#### Де знаходяться файли і як запускається збірка

Тепер, коли ви бачили `COPY . .` у Dockerfile, постає питання: **що саме означає ця перша крапка?** Звідки Docker бере файли?

Відповідь — **Build Context**: директорія на хост-машині, яку Docker використовує як джерело файлів під час збірки. Ви вказуєте її при запуску команди збірки:

```bash
docker build -t my-flask-app .
```

| Частина команди | Що означає                                                                                                   |
| ----------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| `docker build`              | Зібрати Docker-образ із Dockerfile                                                                      |
| `-t my-flask-app`           | Дати образу ім'я (tag). Без `-t` образ матиме лише хеш-ідентифікатор |
| `.`                         | Build context — поточна директорія як джерело файлів                                 |

```
training-project/          ← Build Context (поточна директорія)
├── Dockerfile             ← інструкції для Docker
├── app.py                 ← буде скопійовано в образ
├── requirements.txt       ← буде скопійовано в образ
└── .dockerignore          ← що НЕ копіювати
```

Тепер `COPY . .` читається однозначно:

- Перша крапка `.` — **build context** (`training-project/` на хост-машині)
- Друга крапка `.` — **`WORKDIR` всередині образу** (у нашому випадку `/app`)

Після збірки образ зберігається у внутрішньому сховищі Docker (`/var/lib/docker/overlay2/`) як набір шарів. Ви не бачите його як звичайну директорію — тільки через `docker images`.

#### Чому `COPY requirements.txt` перед `COPY . .`?

Це важлива оптимізація, пов'язана із шарами. Docker кешує кожен шар. Якщо файл не змінився — шар береться з кешу без повторного виконання.

Код змінюється часто. `requirements.txt` — рідко. Якщо ми спочатку копіюємо `requirements.txt` і встановлюємо залежності — цей дорогий крок (`pip install`) кешується. При наступній збірці (після зміни `app.py`) Docker пропустить `pip install` і почне тільки з `COPY . .`.

Якщо ж зробити навпаки — `COPY . .` першим — то будь-яка зміна коду інвалідує кеш `pip install`, і він виконується щоразу заново.

#### `.dockerignore` — що не повинно потрапити в образ

`COPY . .` копіює **все** з поточної директорії в образ. Без фільтра в образ потраплять `.env` (секрети!), `.git/` (історія репозиторію), `venv/` (локальні залежності), `__pycache__/`.

Файл `.dockerignore` працює так само як `.gitignore` з Теми 5: перераховує те, що Docker має ігнорувати під час `COPY`.

```
.env
.git
venv/
__pycache__/
*.pyc
```

Якщо `.gitignore` захищає репозиторій від зайвих файлів, то `.dockerignore` захищає образ — від зайвого розміру і від витоку секретів.

**Ключові поняття:**

- **Dockerfile** — текстовий файл з інструкціями побудови образу; IaC для одного сервісу
- **`FROM`** — базовий образ; точка відліку для нового образу
- **`RUN`** — команда, що виконується під час збірки і фіксується як шар
- **`CMD`** — команда запуску контейнера; на відміну від `RUN`, виконується при кожному старті
- **Layer caching (кешування шарів)** — Docker повторно використовує незмінені шари; визначає оптимальний порядок інструкцій
- **`.dockerignore`** — файл-фільтр, що визначає, які файли не копіювати в образ

### 5. docker-compose — запуск сервісів однією командою

Запускати контейнер командою `docker run` з довгим списком аргументів — незручно і не відтворювано. `docker-compose` вирішує це: описуємо сервіс у файлі `docker-compose.yml` і запускаємо однією командою.

#### Приклад: PostgreSQL для нашого Flask-додатку

Наш Flask-додаток вже працює як systemd-сервіс (Тема 7). Тепер ми додаємо до нього базу даних:

```yaml
services:
  db:
    image: postgres:16
    restart: unless-stopped
    environment:
      POSTGRES_DB: training_db
      POSTGRES_USER: training_user
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
    volumes:
      - postgres_data:/var/lib/postgresql/data
    ports:
      - "5432:5432"

volumes:
  postgres_data:
```

Flask працює через systemd, PostgreSQL — через Docker. Кожен інструмент робить те, в чому він найкращий.

#### Розбираємо ключові концепції

**`image: postgres:16`:**

Використовуємо готовий офіційний образ з Docker Hub. Не потрібно нічого збирати — хтось уже підготував PostgreSQL 16 зі всіма залежностями. Це і є головна перевага Docker для сторонніх сервісів: одна команда замість 10 кроків ручного встановлення.

**`ports` — прокидання портів:**

```yaml
ports:
  - "5432:5432"   # HOST_PORT:CONTAINER_PORT
```

Контейнер живе у своїй ізольованій мережі. Без `ports` його порт недоступний з хост-системи. `"5432:5432"` означає: «порт 5432 на хості → порт 5432 в контейнері».

Це схоже на Port Forwarding з Теми 3, але в межах однієї машини. Flask-додаток, що працює через systemd на VM, підключається до `localhost:5432` — і потрапляє в контейнер з PostgreSQL.

**`volumes` — збереження даних між перезапусками:**

Контейнер за замовчуванням — **ephemeral** (тимчасовий). Всі зміни у файловій системі контейнера зникають після його **видалення** (`docker compose down`). Зверніть увагу: зупинка (`docker compose stop`) зберігає дані, але видалення — ні. Для PostgreSQL це було б катастрофою — база даних зникала б при перестворенні контейнера.

`volumes` вирішує це: директорія `/var/lib/postgresql/data` всередині контейнера монтується з іменованого volume `postgres_data`, який зберігається на хост-системі незалежно від циклу контейнера. Навіть після `docker compose down` дані залишаться (якщо не додати прапорець `-v`).

**Де фізично зберігається volume:**

Named volume `postgres_data` зберігається у спеціальній директорії Docker на хост-системі (VM):

```
/var/lib/docker/volumes/postgres_data/_data/
```

Це **не** у вашій `training-project/`, а в централізованому сховищі Docker. Туди Docker записує всі файли бази даних PostgreSQL. Навіть якщо ви видалите контейнер (`docker compose down`), ця директорія залишиться на місці з усіма даними. Вона зникне тільки при `docker compose down -v` (прапорець `-v` = видалити volumes).

**`restart: unless-stopped` — автоперезапуск контейнера:**

Це аналог `Restart=on-failure` з unit-файлу systemd (Тема 7), але для контейнерів. Docker daemon слідкує за контейнером і перезапускає його при збої. Після перезавантаження VM контейнер теж підніметься автоматично — якщо Docker daemon запущений (а він запущений, бо це systemd-сервіс — див. розділ 7).

| Політика restart | Коли перезапускає                                                                                      |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------- |
| `no`                   | Ніколи (за замовчуванням)                                                                         |
| `on-failure`           | Тільки при аварійному завершенні (ненульовий код виходу)               |
| `unless-stopped`       | Завжди, крім випадку коли контейнер зупинили вручну                        |
| `always`               | Завжди, навіть якщо зупинили вручну (після перезавантаження Docker) |

**`${POSTGRES_PASSWORD}`** — змінна оточення з `.env` файлу (або з середовища). Це та сама практика управління секретами, яку ми запровадили в Темах 5–7: паролі не зберігаються у `docker-compose.yml`, який йде в Git.

#### Для довідки: повна контейнеризація

У нашому курсі Flask працює як systemd-сервіс, а Docker використовується для допоміжних залежностей. Але у виробничих проєктах часто контейнеризують і сам додаток. Тоді `docker-compose.yml` виглядає так:

```yaml
services:
  db:
    image: postgres:16
    restart: unless-stopped
    environment:
      POSTGRES_DB: training_db
      POSTGRES_USER: training_user
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
    volumes:
      - postgres_data:/var/lib/postgresql/data

  app:
    build: .
    restart: unless-stopped
    ports:
      - "5000:5000"
    environment:
      DATABASE_URL: postgresql://training_user:${POSTGRES_PASSWORD}@db:5432/training_db
    depends_on:
      - db

volumes:
  postgres_data:
```

Зверніть увагу: у сервісу `db` тут **немає** `ports: "5432:5432"`. Це не помилка.

У нашому основному прикладі Flask працює **поза** Docker-мережею (як systemd-сервіс на хості), тому PostgreSQL мусить відкрити порт назовні — інакше Flask його не побачить. Звідси `ports: "5432:5432"`.

У цьому варіанті `app` і `db` знаходяться **в одній внутрішній Docker-мережі** і спілкуються напряму: `app` просто звертається до `db:5432`. Порт `5432` нікуди не прокидається — база взагалі недоступна з хост-системи. Це безпечніший підхід: менше відкритих портів.

> **Правило**: `ports` потрібні лише тоді, коли до контейнера має підключитися хтось **поза** Docker-мережею.



У цьому варіанті `build: .` збирає образ із Dockerfile, а `depends_on` гарантує порядок запуску (але не готовність PostgreSQL до з'єднань — лише старт контейнера).

Зверніть увагу на `DATABASE_URL`:

```
postgresql://training_user:...@db:5432/training_db
                               ↑
                    ім'я сервісу з compose-файлу
```

`docker-compose` автоматично створює внутрішню мережу для всіх сервісів у файлі і реєструє кожен сервіс під його іменем. Ім'я `db` — це саме те, що написано зліва від двокрапки у YAML (`db:`). Замість IP-адреси контейнер `app` просто звертається до `db:5432` — Docker вирішує, до якого контейнера це відповідає.

Ми розглянемо цей підхід детальніше в Темі 9 (CI/CD), де автоматизований пайплайн може збирати Docker-образ при кожному push.

**Ключові поняття:**

- **docker-compose** — інструмент для декларативного запуску контейнерів з YAML-файлу
- **`ports`** — прокидання порту: `HOST:CONTAINER`, робить порт контейнера доступним ззовні
- **Volume** — механізм збереження даних поза контейнером; дані переживають видалення контейнера
- **Ephemeral (тимчасовий)** — характеристика контейнера: без volumes дані зникають при видаленні
- **`restart: unless-stopped`** — політика автоперезапуску: Docker daemon слідкує за контейнером і піднімає його після збою або перезавантаження

### 6. Основні команди Docker

#### Команди docker-compose (основні для лабораторної)

```bash
# Запустити всі сервіси у фоні (-d = detached)
docker compose up -d

# Зупинити і видалити контейнери (volumes зберігаються)
docker compose down

# Зупинити і видалити контейнери + видалити volumes (УВАГА: дані бази знищаться)
docker compose down -v

# Переглянути запущені контейнери і їх статус
docker compose ps

# Переглянути логи всіх сервісів
docker compose logs

# Переглянути логи конкретного сервісу в реальному часі
docker compose logs -f db

# Зібрати/перезібрати образ (якщо є сервіси з build:)
docker compose build

# Перезібрати і перезапустити (якщо є сервіси з build:)
docker compose up -d --build
```

#### Команди docker (робота з окремими контейнерами)

```bash
# Переглянути запущені контейнери
docker ps

# Переглянути всі контейнери (включно зі зупиненими)
docker ps -a

# Виконати команду всередині запущеного контейнера
docker exec -it <container_name> bash

# Переглянути логи контейнера
docker logs <container_name>

# Переглянути локально завантажені образи
docker images
```

#### Важлива різниця: `docker compose` vs. `docker-compose`

У сучасних версіях Docker (Docker Desktop та Docker Engine ≥ 20.10) compose є вбудованим плагіном і викликається як `docker compose` (через пробіл). Стара утиліта `docker-compose` (з дефісом) — окремий Python-інструмент, що поступово виходить з ужитку. У лабораторній ми використовуємо сучасний синтаксис: `docker compose`.

**Ключові поняття:**

- **`docker compose up -d`** — запуск у фоні (detached mode)
- **`docker compose down`** — зупинка та видалення контейнерів
- **`docker exec -it`** — запуск інтерактивного процесу всередині контейнера (наприклад, `bash`)
- **`docker compose logs -f`** — потоковий перегляд логів (аналог `journalctl -f`)

### 7. Де Docker вписується у наш стек

#### Хто за що відповідає

Після Теми 8 на нашій VM працюють два механізми управління процесами:

| Що                            | Хто керує       | Як автоматично стартує після перезавантаження VM |
| ------------------------------- | ----------------------- | ----------------------------------------------------------------------------------------- |
| Flask-додаток            | systemd (unit-файл) | `systemctl enable training-app`                                                         |
| Docker daemon                   | systemd                 | `systemctl enable docker`                                                               |
| PostgreSQL (контейнер) | Docker (restart policy) | `restart: unless-stopped` у compose-файлі                                         |

Зверніть увагу на ланцюжок: при завантаженні VM `systemd` піднімає Docker daemon → Docker daemon бачить контейнери з `restart: unless-stopped` → Docker піднімає PostgreSQL. Flask-додаток systemd піднімає напряму. Два шляхи, один результат — всі сервіси працюють без ручного втручання.

#### З'єднуємо всі теми

Ми додаємо новий рівень до стека, побудованого за попередні теми:

**Тема 8: Docker**
`docker-compose.yml` описує і запускає PostgreSQL-контейнер

**Тема 7: systemd**
Flask працює як systemd-сервіс; Docker daemon — теж systemd-сервіс

**Тема 6: Ansible**
Ansible може встановлювати Docker і копіювати `docker-compose.yml` на сервер

**Тема 5: Git**
`docker-compose.yml` — це код; він живе в репозиторії разом з `app.py`

**Тема 4: SSH**
`docker exec` дає доступ всередину контейнера — аналог SSH, але для процесів

**Тема 3: Мережа**
Port forwarding у compose — схожа концепція до Port Forwarding у Vagrantfile

**Тема 2: Linux**
Контейнери використовують механізми ядра (namespaces, cgroups) для ізоляції процесів

**Тема 1: VM**
Ми запускаємо Docker **всередині** VM — це типова схема для розробки і курсових робіт

#### Чому Flask залишається в systemd?

У лабораторній ми контейнеризуємо тільки PostgreSQL. Flask-додаток залишається як systemd-сервіс на VM. Це свідоме рішення:

- Ми **додаємо** інструмент (Docker для залежностей), а не **замінюємо** те, що вже працює (systemd для додатку).
- Студенти бачать **контраст**: коли достатньо systemd (свій код, простий сервіс), а коли Docker дає реальну перевагу (стороння залежність, складне встановлення, потреба в стандартизації).
- У реальних проєктах теж часто зустрічається гібридний підхід — не все живе в контейнерах.

## ❓ Питання для обговорення на занятті

1. У Темі 1 ми обрали VM замість Docker, бо нам потрібен `systemd`. У Темі 8 ми запускаємо Docker **всередині** VM. Чи немає тут суперечності? Чому таке поєднання — нормальна практика, а не «кроки назад»?
2. Контейнер — ephemeral. Після `docker compose down -v` дані PostgreSQL зникнуть. Але після `docker compose down` (без `-v`) — залишаться. Чому? Як volumes вирішують проблему втрати даних?
3. У unit-файлі з Теми 7 є `Restart=on-failure`. У `docker-compose.yml` є `restart: unless-stopped`. Обидва механізми перезапускають процес після збою. Але хто саме слідкує за процесом у кожному випадку? Чому Docker daemon сам потребує systemd для свого запуску?
4. У unit-файлі з Теми 7 є `EnvironmentFile=.env`. У `docker-compose.yml` є `${POSTGRES_PASSWORD}`. Обидва підходи читають секрети з оточення. У чому спільний принцип? Де саме зберігається `.env` у цьому випадку і чому він не має потрапити в Git?
5. Ми запустили PostgreSQL однією командою `docker compose up -d`. Порівняйте це з ручним встановленням: `apt install`, створення користувача, ініціалізація кластера, створення бази. Які переваги та недоліки кожного підходу? Коли ручне встановлення може бути виправданим?
