## 🎯 Мета роботи

Навчитись впевнено орієнтуватись у Linux: досліджувати файлову систему, керувати файлами, переглядати стан системи (RAM, CPU, диск) та керувати правами доступу. На завершення — створити виділеного системного користувача `openclaw` для нашого AI-агента, дотримуючись принципу найменших привілеїв.

> **Контекст:** Ви продовжуєте роботу з VM з Теми 1. Запустіть VM та підключіться до неї.

## 🛠 Покрокова інструкція

### Крок 1: Запускаємо VM та підключаємось

Відкрийте термінал **на вашому комп'ютері (хості)**, перейдіть у теку з Vagrantfile та підключіться:

```bash
# Перейти в теку з Vagrantfile (створена в Темі 1)
cd ~/devops-course/training-project/

# Запустити VM (якщо вона зупинена після минулого заняття)
vagrant up

# Підключитись до VM
vagrant ssh
```

**Очікуваний результат:** Рядок запрошення зміниться на `vagrant@devops-sandbox:~$` — ви всередині VM.

### Крок 2: Хто я і де я знаходжусь?

Перші три команди, які DevOps-інженер виконує на будь-якому незнайомому сервері:

```bash
# Хто я? (поточний користувач)
whoami

# Де я знаходжусь? (поточна тека)
pwd

# Загальна інформація про систему
uname -a
```

**Очікуваний результат:**

```text
vagrant
/home/vagrant
Linux devops-sandbox 5.15.x ... x86_64 GNU/Linux
```

### Крок 3: Досліджуємо файлову систему

```bash
# Перейти в корінь файлової системи
cd /

# Переглянути вміст кореневої директорії
ls -l

# Переглянути ключові директорії
ls /home/       # Домашні теки користувачів
ls /opt/        # Тут буде жити наш агент
ls /etc/        # Конфігурації системи (читати кожен, писати тільки root)
ls /var/log/    # Системні журнали (логи)
ls /tmp/        # Тимчасові файли
```

**Очікуваний результат:** Ви бачите вміст кожної директорії. Порівняйте структуру з таблицею «Linux vs Windows» з теоретичного конспекту.

### Крок 4: Навігація та знайомство з командами

Поверніться в домашню теку і попрактикуйте навігацію:

```bash
# Повернутись у домашню теку (три еквівалентні способи)
cd /home/vagrant
cd ~
cd

# Переглянути вміст, включно з прихованими файлами
ls -la

# Переглянути bashrc (файл налаштувань shell)
cat ~/.bashrc

# Переглянути hostname сервера
cat /etc/hostname
```

**Очікуваний результат:** У списку `ls -la` ви побачите приховані файли (починаються з крапки): `.bashrc`, `.profile`, `.bash_history` тощо.

### Крок 5: Стан системи — RAM, SWAP, CPU та диск

Перевіримо «здоров'я» нашої VM:

```bash
# Оперативна пам'ять та SWAP (h = human-readable: MB, GB)
free -h

# Дисковий простір на всіх розділах
df -h

# Час роботи VM та середнє завантаження
uptime

# Скільки місця займає тека /var/log
du -sh /var/log/
```

**Завдання:** Запишіть або запам'ятайте:

- Загальний обсяг RAM вашої VM: ______
- Чи використовується SWAP зараз? (так / ні): ______
- Скільки вільно місця на диску `/`: ______

```bash
# Список процесів (статичний знімок)
ps aux

# Пошук конкретного процесу (наприклад, bash)
ps aux | grep bash

# Інтерактивний моніторинг (вийти: натисніть q)
top
```

> 💡 У `top`: стовпець `%CPU` — завантаження процесора, `%MEM` — пам'ять. Рядок `load average` — середнє навантаження за 1, 5 та 15 хвилин.

### Крок 6: Робота з файлами та текою

Створимо тренувальну теку та попрактикуємо роботу з файлами:

```bash
# Створити теку для практики
mkdir ~/lab02_practice
cd ~/lab02_practice

# Створити кілька файлів
touch readme.txt notes.txt config.txt

# Переглянути список (з деталями)
ls -la

# Записати текст у файл (> = перезаписати, >> = дописати)
echo "Це мій перший файл" > readme.txt
echo "Рядок 1" > notes.txt
echo "Рядок 2" >> notes.txt

# Переглянути вміст
cat readme.txt
cat notes.txt

# Скопіювати та перейменувати
cp readme.txt readme_backup.txt
mv config.txt settings.txt

# Переглянути результат
ls -la
```

**Очікуваний результат:** У теці `lab02_practice` є файли: `readme.txt`, `readme_backup.txt`, `notes.txt`, `settings.txt`.

### Крок 7: Права доступу — читаємо та змінюємо

```bash
# Переглянути права поточних файлів
ls -la ~/lab02_practice/
```

Розшифруйте перший стовпець. Наприклад, `-rw-r--r--`:

```text
- rw- r-- r--
│ │   │   └── інші: тільки читання
│ │   └── група: тільки читання
│ └── власник: читання + запис
└── тип: - (звичайний файл)
```

```bash
# Зробити файл невидимим для інших (тільки власник читає)
chmod 600 settings.txt
ls -la settings.txt
# Результат: -rw------- vagrant vagrant settings.txt

# Зробити файл публічно читабельним
chmod 644 readme.txt
ls -la readme.txt
# Результат: -rw-r--r-- vagrant vagrant readme.txt

# Спробуємо зробити скрипт та дати права на виконання
echo '#!/bin/bash' > myscript.sh
echo 'echo "Скрипт працює!"' >> myscript.sh
chmod 755 myscript.sh
ls -la myscript.sh
# Результат: -rwxr-xr-x vagrant vagrant myscript.sh

# Запустити скрипт
./myscript.sh
```

**Очікуваний результат:** `Скрипт працює!`

### Крок 8: Змінні оточення

```bash
# Переглянути поточного користувача і домашню теку

# Перевірте: у новій сесії ця змінна зникне
```

### Крок 9: Перевіряємо поточного користувача та sudo

```bash
# Детальна інформація про поточного користувача
id

# Які sudo-команди дозволені (vagrant на нашій VM має повний sudo)
sudo -l

# Перевіримо, що vagrant справді може робити дії root
sudo cat /etc/shadow   # Файл паролів — звичайні користувачі не можуть читати
```

**Очікуваний результат:** `sudo -l` покаже `(ALL : ALL) ALL` — vagrant може все. Це проблема для сервісного процесу.

### Крок 10: Створюємо системного користувача `openclaw`

Тепер найважливіший крок — підготовка облікового запису для нашого агента:

```bash
# Створити системного користувача без домашньої теки і без login shell
sudo useradd \
  --system \
  --no-create-home \
  --shell /usr/sbin/nologin \
  --comment "OpenClaw AI Agent" \
  openclaw

# Перевірити, що користувач створений
id openclaw
grep openclaw /etc/passwd
```

**Очікуваний результат:**

```text
uid=999(openclaw) gid=999(openclaw) groups=999(openclaw)
openclaw:x:999:999:OpenClaw AI Agent:/home/openclaw:/usr/sbin/nologin
```

> 📌 `UID` у вашому випадку може відрізнятись (залежить від системи). Головне — він у діапазоні системних користувачів (< 1000).

---

### Крок 11: Налаштовуємо директорії для агента

```bash
# Створити директорії для агента (в /opt, а не /home — це системний сервіс)
sudo mkdir -p /opt/openclaw
sudo mkdir -p /opt/openclaw/logs
sudo mkdir -p /opt/openclaw/config

# Передати директорії у власність openclaw
sudo chown -R openclaw:openclaw /opt/openclaw

# Поставити коректні права (власник — повний доступ, решта — нічого)
sudo chmod 750 /opt/openclaw

# Перевірити результат
ls -la /opt/
ls -la /opt/openclaw/
```

**Очікуваний результат:**

```text
drwxr-x--- 4 openclaw openclaw 4096 Mar 12 20:00 /opt/openclaw/
```

### Крок 12: Перевіряємо ізоляцію

Перевіримо, що `openclaw` справді обмежений:

```bash
# Спроба увійти як openclaw (нологін shell — буде відмовлено)
su - openclaw
# Очікуваний результат: "This account is currently not available."

# Запустити команду від імені openclaw з sudo vagrant
sudo -u openclaw bash

# Тепер ми запускаємо команди ЯК openclaw (але всередині bash-сесії vagrant)
whoami            # openclaw
id                # uid=999(openclaw) ...

# Спробувати вийти за межі своєї теки
cat /etc/shadow   # Помилка: Permission denied ✅
ls /root/         # Помилка: Permission denied ✅

# Перевірити доступ до власної теки
ls /opt/openclaw/ # Успіх — він може читати свою теку ✅

# Вийти з сесії openclaw
exit
```

**Очікуваний результат:** `openclaw` не може читати `/etc/shadow` та `/root/` — ізоляція працює.

## ❓ Контрольні питання

1. Виконайте `ls -la /opt/openclaw/`. Розшифруйте перший рядок: що означають символи `drwxr-x---`, число `2`, та два поля після нього?
2. Що означає `chmod 600 secret.key`? Хто може читати цей файл, хто — писати?
3. Чому ми створили `openclaw` з прапором `--shell /usr/sbin/nologin`? Що станеться, якщо зловмисник спробує виконати `ssh openclaw@сервер`?
4. `free -h` показує `Swap: 1.0Gi | 800Mi used`. Що ви зробите? Чому це погано?
5. Назвіть різницю між `/home/openclaw/` і `/opt/openclaw/`. Чому агент живе в `/opt`, а не в `/home`?
