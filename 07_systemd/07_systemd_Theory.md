## 🎯 Бізнес-задача

У Темі 6 ми досягли важливого результату: Ansible playbook за одну команду розгортає наш Flask-додаток на VM — встановлює залежності, клонує код з GitHub, створює `.env`. Наприкінці лабораторної ми запустили додаток вручну і навіть перевірили через браузер via SSH-тунель.

Але є одна критична проблема: щойно ми закрили термінал — додаток зупинився.

Уявіть сценарій:

- VM перезавантажилась після оновлень — додаток не піднявся.
- Flask-процес впав через непередбачену помилку — ніхто цього не помітить до наступного дня.
- Студент забув залишити термінал відкритим — і сервіс недоступний.

Це не вигаданий сценарій. Саме так виглядає «робота» сервісу без системи управління процесами: він живе лише поки хтось тримає термінал відкритим.

**Нам потрібно:** щоб додаток запускався автоматично при старті системи, перезапускався після збоїв, писав логи, і щоб усе це можна було контролювати стандартними командами Linux.

**Зв'язок з проєктом:** У Темі 6 реальний Ansible playbook із `devops-ai-assistant/` мав секцію «Systemd Service» — це не краєм-ока пункт, а частина повного деплойменту. Автоматизувати розгортання і не автоматизувати запуск — це лише половина справи. Сьогодні ми закриємо цю прогалину для нашого `training-project`.

## 🧠 Необхідні знання

### 1. Що таке системний процес і чому звичайний запуск не підходить

Коли ми запускали Flask у лабораторній:

```bash
python app.py
```

— це **foreground-процес**. Він прив'язаний до нашого термінального сеансу. Закриваємо термінал — сесія завершується — процес отримує сигнал `SIGHUP` і зупиняється.

Можна обійти це за допомогою `nohup` або `&`, але це не вирішення проблеми:

```bash
nohup python app.py &   # запускає у фоні, але при перезавантаженні — не піднімається
```

**Що нам насправді потрібно:**

| Вимога                           | Чи вирішує `nohup &`? | Чи вирішує systemd? |
| ------------------------------------ | ----------------------------- | ----------------------- |
| Не зупиняться коли закрити термінал  | ✅ Так                       | ✅ Так                 |
| Автозапуск при старті системи        | ❌ Ні                        | ✅ Так                 |
| Автоперезапуск при збої             | ❌ Ні                        | ✅ Так                 |
| Логи                                 | ❌ Тільки у файл              | ✅ Централізовано      |
| Стандартне керування (start/stop)    | ❌ Тільки `kill`             | ✅ Так                 |

**Ключові поняття:**

- **Foreground-процес** — процес, прив'язаний до термінальної сесії; завершується разом з терміналом
- **Daemon (демон)** — фоновий процес, що не прив'язаний до термінальної сесії і виконується незалежно

### 2. systemd — система ініціалізації Linux

**systemd** — це система ініціалізації та менеджер сервісів, яку використовують усі сучасні дистрибутиви Linux (Ubuntu, CentOS, Debian, Fedora та інші). Вона відповідає за:

- **Запуск системи** — першою завантажується ядро Linux, потім systemd (PID 1) запускає всі інші процеси
- **Управління сервісами** — запуск, зупинка, перезапуск демонів
- **Логування** — централізований збір логів через `journald`
- **Залежності між сервісами** — «спочатку запусти мережу, потім — базу даних, а лише потім — додаток»

> **PID 1** — перший процес, який запускається в Linux. Усі інші процеси є нащадками PID 1. Це спеціальна роль: якщо PID 1 завершиться — система завершується.

**Чому це важливо для DevOps:**

У Темі 1 ми обрали VM на базі Ubuntu саме тому, що нам потрібен повноцінний `systemd`. Більшість виробничих серверів (VPS, хмарні інстанси) теж використовують Ubuntu або CentOS з systemd. Те, що ми робимо на VM, — те саме, що ви будете робити на реальному Production-сервері.

**Ключові поняття:**

- **systemd** — менеджер системи та сервісів у сучасних дистрибутивах Linux (PID 1)
- **PID 1** — перший процес Linux; systemd виконує цю роль і запускає всі інші процеси

### 3. Unit-файли: мова опису сервісів

У systemd кожен сервіс описується через **unit-файл** — текстовий конфігураційний файл у форматі INI.

Подивіться на паралель з попередньою темою:

| Тема 6 (Ansible)                            | Тема 7 (systemd)                              |
| ------------------------------------------- | --------------------------------------------- |
| `playbook.yml` — описує стан сервера        | `*.service` — описує стан сервісу             |
| Декларативний: «що має бути»                | Декларативний: «як має працювати цей сервіс»  |
| Ansible виконує playbook → бажаний стан     | systemd читає unit → запускає/контролює сервіс|

Це та сама ідея: **декларативний опис бажаного стану**, тільки на іншому рівні абстракції.

#### Структура unit-файлу

Типовий `.service` файл складається з трьох секцій:

```ini
[Unit]
Description=...        # Опис сервісу
After=...              # Після якого сервісу запускати

[Service]
Type=...               # Тип сервісу
User=...               # Від якого користувача запускати
WorkingDirectory=...   # Робоча директорія
ExecStart=...          # Команда для запуску
Restart=...            # Коли перезапускати

[Install]
WantedBy=...           # До якого target підключити (де в процесі завантаження)
```

#### Конкретний приклад для нашого Flask-додатку

```ini
[Unit]
Description=Training Flask Application
After=network.target

[Service]
Type=simple
User=training
WorkingDirectory=/opt/training-app
EnvironmentFile=/opt/training-app/.env
ExecStart=/opt/training-app/venv/bin/python app.py
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

**Розберемо кожну директиву:**

| Директива          | Що означає                                                                                                        |
| ------------------ | ----------------------------------------------------------------------------------------------------------------- |
| `After=network.target` | Запускати лише після того, як мережа готова. Для Flask це потрібно, щоб сокет міг відкритись |
| `Type=simple`      | Основний процес — той, що запустила `ExecStart`. Найпростіший і найпоширеніший тип                               |
| `User=training`    | Запускати від імені `training`, а не root. Принцип найменших привілеїв з Теми 2                                  |
| `WorkingDirectory` | Де знаходяться файли додатку. `app.py`, `venv/` — у цій директорії                                               |
| `EnvironmentFile`  | Завантажити змінні оточення з `.env` файлу. `PORT=5000` стане доступним для `app.py`                            |
| `ExecStart`        | Команда для запуску. Повний шлях до Python із virtualenv — той самий, що ми перевіряли вручну в Темі 6           |
| `Restart=on-failure` | Перезапускати, якщо процес завершився з помилкою (ненульовий код виходу). Не перезапускати при `systemctl stop` |
| `RestartSec=5`     | Чекати 5 секунд перед перезапуском (щоб не «молотити» при постійній помилці)                                     |
| `WantedBy=multi-user.target` | Запускати при досягненні multi-user стану системи (нормальна робота, синонім «системи завантажились») |

**Ключові поняття:**

- **Unit-файл** — текстовий файл конфігурації сервісу для systemd
- **[Unit]** — секція з описом та залежностями
- **[Service]** — секція з параметрами запуску процесу
- **[Install]** — секція з тим, коли і куди підключити сервіс
- **target** — стан системи в systemd; `multi-user.target` ≈ «система повністю завантажена»

### 4. Основні команди systemctl

**systemctl** — командний інтерфейс для управління systemd та сервісами.

#### Повний цикл роботи з сервісом

```bash
# Перезавантажити конфігурацію systemd (після будь-яких змін unit-файлу)
sudo systemctl daemon-reload

# Активувати автозапуск при старті системи
sudo systemctl enable training-app

# Запустити сервіс зараз
sudo systemctl start training-app

# Перевірити статус
sudo systemctl status training-app

# Зупинити сервіс
sudo systemctl stop training-app

# Перезапустити сервіс
sudo systemctl restart training-app

# Деактивувати автозапуск
sudo systemctl disable training-app
```

#### Відмінність між `enable` та `start`

Це часта точка плутанини:

| Команда  | Що робить                                                     | Коли ефект настає         |
| -------- | ------------------------------------------------------------- | ------------------------- |
| `start`  | Запускає сервіс **прямо зараз**                               | Відразу                   |
| `enable` | Реєструє автозапуск **при наступному завантаженні** системи   | Після перезавантаження    |
| `stop`   | Зупиняє сервіс **прямо зараз**                                | Відразу                   |
| `disable`| Скасовує автозапуск **при наступному завантаженні** системи   | Після перезавантаження    |

Щоб сервіс і запустився зараз, і автоматично піднімався після перезавантаження — потрібні обидві команди: `enable` + `start`.

#### Що показує `systemctl status`

```
● training-app.service - Training Flask Application
     Loaded: loaded (/etc/systemd/system/training-app.service; enabled; ...)
     Active: active (running) since Sun 2026-04-06 10:00:00 UTC; 2min ago
   Main PID: 1234 (python)
      Tasks: 1 (limit: 2301)
     Memory: 25.1M
        CPU: 0.542s
     CGroup: /system.slice/training-app.service
             └─1234 /opt/training-app/venv/bin/python app.py

Apr 06 10:00:00 devops-sandbox systemd[1]: Started Training Flask Application.
```

- **`Loaded: enabled`** — автозапуск активований (`enable`). Якщо `disabled` — при перезавантаженні сервіс не підніметься автоматично.

- **`Active: running`** — сервіс зараз працює. Інші можливі стани: `inactive (dead)` — зупинений, `failed` — впав з помилкою.

- **`Main PID: 1234`** — PID основного процесу. Можна використати для `kill` або для пошуку процесу через `ps`.

- **`Memory: 25.1M`** — поточне використання пам'яті. Корисно для діагностики витоків пам'яті або надмірного споживання.

- **`CPU: 0.542s`** — скільки секунд процесор фактично виконував код цього сервісу з моменту його запуску. Це не відсоток завантаження CPU прямо зараз, а лічильник: «за весь час роботи процесор витратив на цей сервіс 0.542 секунди». Якщо значення стрімко зростає — сервіс споживає багато CPU.

- **`CGroup: ...`** — Control Group, системний механізм ізоляції ресурсів. Показує дерево процесів, що належать до цього сервісу. У нашому випадку — один процес Python з PID 1234.

- **Рядки журналу внизу** — останні записи з `journald` для цього сервісу. Те саме, що `journalctl -u training-app -n 10`. Зручно для швидкої діагностики без окремої команди.

**Ключові поняття:**

- **systemctl** — інструмент для управління сервісами та systemd
- **daemon-reload** — команда для застосування змін у unit-файлах без перезавантаження системи
- **enable** — реєстрація сервісу для автозапуску; **start** — негайний запуск
- **disable** — скасування автозапуску; **stop** — негайна зупинка

### 5. Логування: journalctl

Коли процес запускається через systemd, всі його виводи (`stdout` та `stderr`) автоматично потрапляють у централізовану систему логів — **journald**.

Це принциповий крок вперед порівняно з ручним запуском у терміналі, де виводи просто зникали після закриття сесії.

#### Основні команди journalctl

```bash
# Переглянути всі логи сервісу
sudo journalctl -u training-app

# Останні 50 рядків логів
sudo journalctl -u training-app -n 50

# Спостерігати за логами в реальному часі (як tail -f)
sudo journalctl -u training-app -f

# Логи з конкретного часу
sudo journalctl -u training-app --since "2026-04-06 10:00:00"

# Логи за останню годину
sudo journalctl -u training-app --since "1 hour ago"
```

#### Що видно в логах

```
Apr 06 10:00:00 devops-sandbox python[1234]: * Serving Flask app 'app'
Apr 06 10:00:00 devops-sandbox python[1234]: * Running on http://127.0.0.1:5000
Apr 06 10:05:33 devops-sandbox systemd[1]: training-app.service: Main process exited, code=killed, status=9/KILL
Apr 06 10:05:33 devops-sandbox systemd[1]: training-app.service: Failed with result 'signal'.
Apr 06 10:05:38 devops-sandbox systemd[1]: training-app.service: Scheduled restart job...
Apr 06 10:05:38 devops-sandbox systemd[1]: Started Training Flask Application.
Apr 06 10:05:38 devops-sandbox python[1237]: * Serving Flask app 'app'
```

Зверніть: ми бачимо не лише виводи самого Python-процесу, а й дії systemd — коли він виявив збій, скільки чекав і коли перезапустив. Це цінна інформація для діагностики.

**Ключові поняття:**

- **journald** — централізована служба логування, частина systemd
- **journalctl** — інструмент для читання логів journald
- **`-u`** — фільтр за назвою unit; **`-f`** — режим реального часу; **`-n`** — кількість останніх рядків

### 6. Безпека на рівні сервісу

systemd не просто запускає процеси — він може **обмежити** їхні можливості. Це продовження принципу найменших привілеїв з Теми 2, але тепер на рівні ізоляції процесу.

#### Що ми вже маємо

У нашому unit-файлі є два важливі налаштування безпеки:

**1. `User=training`** — запуск від непривілейованого користувача

Якщо зловмисник знайде вразливість у нашому Flask-додатку і зможе виконати довільний код, — він отримає лише права користувача `training`. Без `sudo`, без доступу до системних директорій. Ми вже обмежили карту ризику ще в Темі 2, і зараз знову скористались цим рішенням.

**2. `EnvironmentFile` замість ENV у коді** — секрети не «зашиті» в код

Відповідно до першого правила управління секретами: конфігурація повинна читатись з оточення, а не жити в коді. `.env` файл створений Ansible (Тема 6) з правами `0600` — тільки `training` може читати. `systemd` передає ці змінні у процес через `EnvironmentFile`.

#### Додаткові директиви ізоляції

Для ознайомлення — реальний виробничий unit-файл може містити:

```ini
[Service]
...
ProtectSystem=strict          # файлова система тільки для читання (крім /proc, /dev)
PrivateTmp=true               # ізольований /tmp для цього сервісу
NoNewPrivileges=true          # заборона підвищення привілеїв
```

Ці директиви ми не будемо використовувати в лабораторній, але важливо знати: systemd — це не просто «запускач процесів», а компонент безпеки.

**Ключове розуміння:** кожен рівень нашого стека — Тема 2 (Linux-користувач), Тема 6 (права на файли в Ansible), Тема 7 (unit-файл) — додає ще один шар захисту. Це і є **defense in depth** — ешелонована оборона.

**Ключові поняття:**

- **`User=`** — директива unit-файлу для запуску сервісу від непривілейованого користувача
- **`EnvironmentFile=`** — завантаження змінних оточення з файлу без вбудовування секретів у unit
- **`ProtectSystem=`**, **`PrivateTmp=`**, **`NoNewPrivileges=`** — директиви ізоляції сервісу (для довідки)
- **Defense in depth** — принцип ешелонованого захисту: кілька незалежних рівнів безпеки

**Тема 3: Мережа**  
Сервіс слухає на 127.0.0.1, а не на 0.0.0.0 (ізоляція)

**Тема 2: Linux**  
Окремий користувач training, принцип найменших привілеїв

Ізольоване середовище для безпечних експериментів
