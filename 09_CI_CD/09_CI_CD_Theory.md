Але є одна проблема: **кожного разу, коли ми вносимо зміни до коду, потрібно виконувати ручний ланцюжок дій:**

1. Написали зміну на локальній машині.
2. `git push` до GitHub.
3. Зайшли на сервер через SSH.
4. `git pull` на сервері.
5. Перезапустили systemd-сервіс.

Це чотири ручні кроки після кожної зміни. Тепер уявіть, що у вас не один розробник, а п'ять. Або що оновлення виходять двічі на день. Хто гарантує, що кожна зміна пройшла перевірку перед тим, як потрапити на сервер?

**Нам потрібен конвеєр, який автоматично перевіряє кожну зміну коду і розгортає її на сервер — без нашого втручання.**

**Зв'язок з проєктом:** У `devops-ai-assistant/` є Ansible playbook для розгортання сервісу — він запускається вручну з локальної машини. Папка `.github/workflows/` при цьому порожня. Це типова ситуація на початку впровадження DevOps: автоматизація конфігурації є, автоматизація деплою — ще ні.

## 🧠 Необхідні знання

### 1. Що таке CI та CD — і навіщо їх розділяють

**CI (Continuous Integration — Безперервна інтеграція)** — практика, за якою кожна зміна коду автоматично перевіряється: запускаються тести, перевіряється синтаксис, аналізується якість. Мета — виявляти помилки якомога раніше, до того як вони потраплять на сервер.

**CD (Continuous Delivery / Deployment — Безперервне доставлення / розгортання)** — практика, за якою перевірений код автоматично доставляється до Production-середовища. Різниця між Delivery і Deployment:

- **Continuous Delivery**: код готовий до деплою в будь-який момент, але розгортання відбувається після ручного підтвердження («кнопка деплою»).
- **Continuous Deployment**: розгортання відбувається **автоматично** після успішного проходження всіх перевірок.

```text
Розробник → git push → [CI: перевірки] → [CD: деплой] → Production
                              ↑
                     якщо хоча б одна
                     перевірка впала —
                     деплою немає
```

**Ключова ідея:** CI/CD не усуває помилки — він виявляє їх швидше і не пускає до Production. Якщо Тема 6 (Ansible) навчила нас **як** розгортати відтворювано, то CI/CD навчає нас **коли** і **за якою умовою** це робити.

**Ключові поняття:**

- **CI (Continuous Integration)** — автоматична перевірка кожної зміни коду
- **CD (Continuous Delivery/Deployment)** — автоматичне або напівавтоматичне розгортання перевіреного коду
- **Pipeline (конвеєр)** — послідовність автоматизованих кроків, якими проходить кожна зміна коду

### 2. Анатомія CI/CD пайплайну

Пайплайн складається з **етапів (stages)**, кожен з яких містить **завдання (jobs)**, а кожне завдання — з **кроків (steps)**.

```text
Пайплайн
├── Stage: CI
│   ├── Job: lint         ← перевірка синтаксису
│   └── Job: test         ← автоматичні тести
└── Stage: CD
    └── Job: deploy       ← деплой на сервер
                             (лише якщо CI пройшов)
```

**Типові етапи:**

| Етап     | Що відбувається                                       | Приклад інструментів      |
| -------- | ----------------------------------------------------- | ------------------------- |
| `lint`   | Перевірка синтаксису та стилю коду                    | `flake8`, `ansible-lint`  |
| `test`   | Запуск автоматичних тестів                            | `pytest`, `unittest`      |
| `build`  | Збірка артефакту (Docker-образ, архів, бінарний файл) | `docker build`            |
| `deploy` | Розгортання на сервер                                 | `ansible-playbook`, `ssh` |

**Важливе правило:** кожен наступний етап запускається **тільки якщо попередній успішно завершився**. Якщо `lint` виявив помилку — `deploy` не відбудеться. Це і є головний захисний механізм CI/CD.

**Ключові поняття:**

- **Stage** — логічна група завдань (наприклад, «перевірки», «деплой»)
- **Job** — конкретне завдання, яке виконується в ізольованому середовищі
- **Step** — один крок всередині job (команда або дія)
- **Артефакт** — результат збірки, який передається між етапами

### 3. GitHub Actions — конкретна реалізація

Існують десятки CI/CD платформ: GitHub Actions, GitLab CI, Jenkins, CircleCI. Ми розглянемо **GitHub Actions** — тому що він вбудований у GitHub (наш Git-хостинг з Теми 5), безкоштовний для відкритих репозиторіїв і не потребує окремого сервера.

**Як це працює:**

1. У репозиторії в теці `.github/workflows/` створюємо YAML-файл.
2. GitHub стежить за подіями (push, pull request, schedule).
3. Коли подія відбувається — GitHub запускає **runner** (тимчасову VM), виконує кроки і повертає результат.

```text
.github/
└── workflows/
    └── ci-cd.yml   ← тут живе наш пайплайн
```

**Анатомія workflow-файлу:**

```yaml
name: CI/CD Pipeline          # Назва пайплайну

on:                            # Коли запускати
  push:
    branches: [main]           # при push у гілку main
  pull_request:                # при відкритті PR

jobs:                          # Завдання

  lint:                        # Назва job
    runs-on: ubuntu-latest     # На якому runner запускати
    steps:
      - uses: actions/checkout@v4      # Крок: клонуємо репозиторій
      - name: Install ansible-lint
        run: pip install ansible-lint  # Крок: команда shell
      - name: Run lint
        run: ansible-lint ansible/     # Крок: запускаємо перевірку

  deploy:
    runs-on: ubuntu-latest
    needs: lint                # Цей job запускається ТІЛЬКИ після lint
    if: github.ref == 'refs/heads/main'  # ТІЛЬКИ для гілки main
    steps:
      - uses: actions/checkout@v4
      - name: Deploy via Ansible
        run: ansible-playbook -i inventory ansible/deploy.yml
```

**Ключові поняття:**

- **Workflow** — весь YAML-файл; описує, коли і що запускати
- **Runner** — тимчасова VM, де виконуються кроки
- **`on:`** — тригери; визначають, які події запускають пайплайн
- **`needs:`** — залежності між jobs; `deploy` виконується лише після `lint`
- **`uses:`** — підключення готового Action з GitHub Marketplace

### 4. Лінтери та автоматичні тести — перша лінія захисту

До того як код потрапить на сервер, він повинен пройти два типи перевірок.

**Лінтер (linter)** — інструмент статичного аналізу, який перевіряє код **не запускаючи його**. Він знаходить:

- Синтаксичні помилки (забута дужка, невірний відступ у YAML)
- Порушення стилю коду (занадто довгий рядок, невикористана змінна)
- Потенційно небезпечні конструкції

Для нашого стеку:

- `flake8` або `ruff` — для Python-коду
- `ansible-lint` — для Ansible playbooks (перевіряє YAML-структуру, знаходить типові антипатерни)

**Автоматичні тести** — код, який запускає ваш код і перевіряє, що він поводиться правильно. Для нашого Flask-додатку це може бути перевірка того, що `/health` повертає `{"status": "ok"}`.

```python
# tests/test_app.py
def test_health_endpoint(client):
    response = client.get("/health")
    assert response.status_code == 200
    assert response.json["status"] == "ok"
```

**Чому це важливо до деплою?** Розглянемо реальний сценарій: якщо Ansible playbook містить синтаксичну помилку у YAML — під час наступного деплою він зламає конфігурацію сервера. `ansible-lint` знаходить цю помилку за секунди, ще на етапі Pull Request — до того як код потрапив у `main`.

**Ключові поняття:**

- **Лінтер** — інструмент статичного аналізу коду без його виконання
- **Unit-тест** — тест окремої функції або компонента в ізоляції
- **Статичний аналіз** — перевірка коду без запуску

### 5. Безпечне зберігання секретів у CI/CD

Для деплою на сервер нам потрібен **SSH-ключ** (той самий, що ми налаштували в Темі 4). Але ми не можемо покласти приватний ключ прямо в YAML-файл у репозиторії — це зробить його публічним.

```yaml
# ТАК РОБИТИ НЕ МОЖНА — ключ стане публічним!
- name: Deploy
  run: |
    echo "-----BEGIN OPENSSH PRIVATE KEY-----
    b3BlbnNzaC1rZXktdjEAAAAA...
    -----END OPENSSH PRIVATE KEY-----" > ~/.ssh/id_ed25519
```

#### Рішення: Secrets (Секрети) у GitHub Actions

GitHub надає захищене сховище для конфіденційних даних. Секрети:

- Шифруються і зберігаються окремо від коду
- Доступні в пайплайні як змінні оточення
- **Ніколи не виводяться у логи** (навіть якщо спробувати `echo $SECRET`, GitHub приховає значення)
- Доступні тільки авторизованим учасникам репозиторію

**Як використовувати секрети:**

1. У налаштуваннях репозиторію: `Settings → Secrets and variables → Actions`.
2. Додаємо секрет з іменем, наприклад `SSH_PRIVATE_KEY`.
3. У workflow посилаємось через `${{ secrets.SSH_PRIVATE_KEY }}`.

```yaml
- name: Setup SSH
  run: |
    mkdir -p ~/.ssh
    echo "${{ secrets.SSH_PRIVATE_KEY }}" > ~/.ssh/id_ed25519
    chmod 600 ~/.ssh/id_ed25519
    ssh-keyscan -H ${{ secrets.SERVER_IP }} >> ~/.ssh/known_hosts
```

**Які секрети потрібні для нашого сценарію:**

Пригадайте архітектуру з попередніх тем: на сервері є користувач `vagrant` (з sudo-правами) для адміністрування і `training` (без sudo) для запуску сервісу. Для CI/CD підключається саме `vagrant` — бо йому потрібно запустити Ansible.

| Секрет            | Що зберігає                                          |
| ----------------- | ---------------------------------------------------- |
| `SSH_PRIVATE_KEY` | Приватний SSH-ключ для підключення до сервера        |
| `SERVER_IP`       | IP-адреса сервера (для студентів — `192.168.56.10`)  |
| `SERVER_USER`     | Ім'я користувача на сервері (`vagrant`)              |

**Ключові поняття:**

- **Secrets** — захищене сховище конфіденційних даних у CI/CD платформі
- **`${{ secrets.NAME }}`** — синтаксис доступу до секрету в GitHub Actions
- **Змінна оточення (environment variable)** — спосіб передачі конфігурації в процес без кодування у файлі

### 6. Ansible у CI/CD — з'єднання всього разом

У Темі 6 ми писали Ansible playbook і запускали його вручну: `ansible-playbook -i inventory.ini playbook.yml`. Тепер цю саму команду виконуватиме CI/CD-runner автоматично.

**Чому Ansible добре вписується в CI/CD:**

1. **Ідемпотентність** (яку ми вивчили в Темі 6): можна запускати playbook хоч після кожного коміту — він застосує лише нові зміни.
2. **Без агента**: Ansible підключається до сервера через SSH — той самий механізм, що ми налаштували в Темі 4.
3. **Одна команда**: замість чотирьох ручних кроків — один виклик `ansible-playbook`.

**Що відбувається під час автоматичного деплою:**

```text
git push (в main)
    ↓
GitHub Actions runner запускається на ubuntu-latest
    ↓
ansible-lint ansible/ → перевірка синтаксису playbook
    ↓ (якщо OK)
ansible-playbook -i inventory.ini playbook.yml
    ↓
Ansible підключається до сервера через SSH
(використовуючи ключ із Secrets)
    ↓
git pull на сервері → перезапуск systemd-сервісу
    ↓
Новий код на сервері
```

**Студентська практика vs реальний проєкт:**

У лабораторній роботі студенти деплоять на свою Vagrant VM (`192.168.56.10`) — це безкоштовно та не потребує VPS. Концепція та команди — ідентичні до Production-сценарію.

У реальних проєктах картина аналогічна: GitHub Actions запускає `ansible-lint` на playbook, а після успішної перевірки — `ansible-playbook`, який підключається до сервера через SSH і перезапускає systemd-сервіс. Різниця лише в адресі сервера — `192.168.56.10` у студента, публічний IP у production. Підхід — той самий.

**Практичне правило:** CI-перевірки (`lint`, `test`) запускаються при **будь-якому** push і при **відкритті PR** — щоб зловити помилку до злиття. Деплой (`deploy`) запускається **тільки при push у `main`** — щоб не деплоїти кожну feature-гілку.

### 7. Гілкова стратегія та CI/CD

CI/CD не існує у вакуумі — він тісно пов'язаний із тим, як команда працює з гілками в Git (Тема 5).

**Типовий захищений workflow:**

```text
feature/update-ansible-playbook
    ↓ (git push)
    CI: ansible-lint → OK
    ↓
    Відкриваємо Pull Request до main
    ↓
    CI запускається ще раз: lint + test → OK
    ↓ (PR затверджений, злиття в main)
    CD: автоматичний ansible-playbook на сервер
```

**Захист гілки `main`** (Branch Protection Rules у GitHub):

- Заборонити прямий push у `main` — тільки через PR.
- Вимагати, щоб CI пройшов перед злиттям.
- Вимагати перевірку (code review) хоча б одного учасника команди.

Це означає: **в `main` потрапляє тільки код, який пройшов автоматичні перевірки.** І тільки такий код деплоїться на сервер.

**Ключові поняття:**

- **Pull Request (PR)** — запит на злиття гілки; місце для обговорення змін і запуску CI
- **Branch Protection** — правила, що захищають гілку від прямих змін без перевірок
- **Merge** — злиття гілки в `main` після проходження всіх умов

## ❓ Питання для обговорення на занятті

1. Ми витратили кілька тем на ручне налаштування середовища. Чому CI/CD вивчається лише зараз, а не раніше — у Темі 2 чи Темі 4?

2. Уявіть: розробник запушив код, CI пройшов успішно, деплой відбувся автоматично. Через годину виявляється, що новий код має логічну помилку (не синтаксичну). Як CI/CD допомагає або не допомагає в цій ситуації?

3. Чому приватний SSH-ключ не можна закомітити в репозиторій у зашифрованому вигляді замість використання Secrets?

4. В яких ситуаціях **Continuous Delivery** (деплой після ручного підтвердження) краще підходить, ніж **Continuous Deployment** (повністю автоматичний деплой)?

5. Ansible забезпечує **ідемпотентність**. Поясніть, чому ця властивість особливо важлива саме в контексті CI/CD, де playbook може запускатися кілька разів на день.
